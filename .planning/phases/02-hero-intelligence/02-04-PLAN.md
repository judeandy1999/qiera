    - Clicking the backdrop (a click whose target is the `<dialog>` element itself) calls `onClose`.
    - When the modal closes, focus returns to the element that was focused immediately before it opened (the trigger).

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser DOM ↔ client modal island | DetailModal runs client-side JS (dialog control + focus management) but takes only serializable, developer-controlled props + children |
| child content → modal shell | Arbitrary `children` are rendered inside the dialog; the shell itself adds no HTML-from-string |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-08 | Denial of Service | focus trap / repeated open-close | mitigate | Uses the browser-native `<dialog>` focus trap (no hand-rolled loop that could spin); open/close is idempotent (guards `dialog.open` before showModal/close). |
| T-02-09 | Injection (XSS) | rendered `children` | mitigate | The shell renders `children` as React nodes and never uses `dangerouslySetInnerHTML`; content sanitization is inherent to JSX escaping (content authored in typed data). |
| T-02-10 | Tampering (focus theft / UI redress) | modal backdrop over page | accept | Native `<dialog>` top-layer rendering + inert background is the standard mitigation; site-level `frame-ancestors` is a Phase 10/deployment concern. |
| T-02-SC | Tampering | npm/pip/cargo installs | mitigate | No packages installed (native `<dialog>` + existing testing libs); no legitimacy gate required. |
</threat_model>
